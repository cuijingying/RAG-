# 培养方案智能问答系统 — 项目完整介绍

## 1. 项目背景与目标

本项目是上海财经大学《非结构化数据处理》课程的期末 Project（Option 1：工程应用路径）。

### 1.1 项目定位

基于课程所学的大模型和 NLP 技术，开发一个**可运行、可复现、可容器化部署**的 RAG 智能问答系统。

### 1.2 核心目标

让学生可以用**自然语言**查询软件工程硕士的培养方案信息，无需手动翻阅文档。系统支持：

- 自然语言查询课程信息、学分要求、修读规则等
- 基于语义理解的智能检索
- 准确、实时的流式问答响应
- 友好的 Streamlit 交互界面
- Docker 容器化一键部署

### 1.3 数据源

核心数据：`program.md` —— 上海财经大学软件工程专业硕士研究生培养方案（2025版）

包含内容：

- **开课院系**：计算机与人工智能学院
- **专业**：软件工程专业
- **学生类别**：硕士研究生（普通研究生）
- **学制**：3 年
- **总学分要求**：37 学分

文档结构涵盖 9 大模块：

1. 学位公共课（5 学分）
2. 学位基础课（9 学分）
3. 专业必修课（8 学分）
4. 专业选修课（6 学分）
5. 国际课程（至少 3 学分）
6. 任意选修课（至少 4 学分）
7. 实践教学环节（2 学分）
8. 其它必修环节（不计学分，硬性通过指标）
9. 毕业总计要求（37 学分）

## 5. 关键技术实现

### 5.1 文档切片算法（ingest.py）

**选型**：MarkdownHeaderTextSplitter

```python
headers_to_split_on = [("##", "Header 2")]
markdown_splitter = MarkdownHeaderTextSplitter(headers_to_split_on=headers_to_split_on)
docs = markdown_splitter.split_text(markdown_text)
```

**特点**：

- 按二级标题（##）智能切片
- 保留文档语义结构
- 自动提取元数据（Header 2）
- 共生成 10 个文档片段

### 5.2 向量嵌入算法（ingest.py）

**选型**：BAAI/bge-small-zh-v1.5

```python
embeddings = HuggingFaceEmbeddings(
    model_name="BAAI/bge-small-zh-v1.5",
    model_kwargs={'device': 'cpu'},
    encode_kwargs={'normalize_embeddings': True}
)
```

**特点**：

- 专为中文语义检索优化
- 轻量级（约 400MB）
- 支持向量归一化

### 5.3 语义检索（app.py）

```python
vectorstore = load_vector_store()
retriever = vectorstore.as_retriever(search_kwargs={"k": 3})
docs = retriever.invoke(prompt)
```

**特点**：

- 余弦相似度语义搜索
- 检索 Top-3 最相关文档片段
- 首次加载后缓存，后续秒级响应

### 5.4 流式问答（app.py）

```python
response = llm.stream(formatted_prompt)
for chunk in response:
    if chunk.content:
        full_response += chunk.content
        message_placeholder.markdown(full_response + "▌")
```

**特点**：

- stream=True 逐 token 实时返回
- 打字机效果，提升用户体验
- 支持对话历史记录

### 5.5 Prompt 工程设计（app.py）

```python
template = """你是一个专业的学术顾问...
请根据以下检索到的相关文档片段，准确回答用户的问题。
【检索到的相关文档片段】：{context}
【用户问题】：{question}
回答要求：
1. 基于提供的文档片段回答，不要编造信息
2. 如果文档中没有相关信息，请明确说明
3. 回答要准确、清晰、有条理
4. 可以适当引用具体的课程代码、学分、学期等信息
5. 使用友好、专业的语气

请开始回答："""
```

**设计原则**：

- 角色设定：专业学术顾问
- 上下文注入：检索到的文档片段
- 约束条件：禁止编造，明确拒绝无信息问题
- 输出规范：引用具体数据，友好专业

### 5.6 缓存机制（app.py）

```python
@st.cache_resource
def get_embeddings():
    return HuggingFaceEmbeddings(...)

@st.cache_resource
def load_vector_store():
    return Chroma(persist_directory="./chroma_db", embedding_function=get_embeddings())
```

**优势**：

- 模型只加载一次，后续请求复用
- 避免重复构建向量数据库
- 大幅提升响应速度

## 8. 技术亮点

### 8.1 RAG 架构

基于检索增强生成（Retrieval-Augmented Generation）技术，将检索的准确性与生成的灵活性结合：

- **检索**：从向量数据库中找出最相关的文档片段
- **增强**：将检索结果注入 Prompt 作为上下文
- **生成**：大模型基于上下文生成准确回答

### 8.2 流式输出

与传统的一次性返回不同，系统采用流式输出（SSE），用户可以看到 AI 逐字生成回答，体验更流畅自然。

### 8.3 缓存优化

通过 `@st.cache_resource` 装饰器，将 Embedding 模型和向量数据库加载到缓存中，避免重复加载，大幅提升响应速度。

### 8.4 容器化部署

完整的 Dockerfile 配置，支持一键构建和运行，确保环境一致性和可复现性。

## 9. 局限性与改进方向

### 9.1 当前局限性

- **Embedding 模型**：使用开源中文模型，语义理解能力有限
- **知识库规模**：仅包含 1 个培养方案文档
- **多轮对话**：不支持复杂的多轮上下文对话
- **并发能力**：Streamlit 的并发处理能力有限

### 9.2 改进方向

- **检索优化**：引入重排序（Re-ranking）机制
- **模型升级**：使用更大的中文语义模型（如 bge-large-zh）
- **知识扩展**：支持多个培养方案的统一检索
- **性能优化**：增加查询结果缓存，减少重复计算
